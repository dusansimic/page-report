# page-report — development guide

Self-hosted service for publishing static HTML report pages generated by AI
agents. Agents upload via the `pr` CLI (ConnectRPC/JSON, OAuth device-flow
bearer token); humans view pages on a separate auth-gated domain. One Go
module, two binaries.

## Repo map

| Path | What lives here |
|---|---|
| `cmd/pr-server` | server entrypoint (wiring only) |
| `cmd/pr` | CLI entrypoint (cobra commands) |
| `proto/pagereport/v1` | protobuf API contract (source of truth) |
| `gen/` | buf-generated code — **never hand-edit** |
| `internal/config` | viper config: YAML + `PR_*` env overrides |
| `internal/auth` | goth web login, session cookie, token validators, allowlist |
| `internal/store` | SQLite persistence + golang-migrate runner |
| `internal/server` | HTTP surface: host-routed muxes, RPC impl, page serving |
| `internal/client` | CLI-side: device flow, credential store, connect client |
| `web/static` | embedded homepage assets |
| `migrations/` | numbered SQL migration pairs, embedded into the binary |
| `skill/SKILL.md` | agent skill teaching the upload flow |

Package boundaries: `internal/client` is imported only by `cmd/pr`;
`internal/server` and `internal/store` only by `cmd/pr-server`.
`internal/config` and `internal/auth` are shared.

## Contract-first workflow

1. Edit `proto/pagereport/v1/pagereport.proto`.
2. `buf lint && buf generate`.
3. Commit `gen/` together with the proto change.

## Migrations

Add a numbered pair `migrations/00000N_name.{up,down}.sql` (golang-migrate
convention). Never edit an already-applied migration. Migrations are embedded
via the `migrations` package and applied automatically at server startup.

## Config

Every key must exist in three places, added together:
`Config` struct + `keys` list in `internal/config/config.go`, and
`config.example.yml`. Env override is `PR_<KEY>` with dots → underscores
(`oidc.client_id` → `PR_OIDC_CLIENT_ID`).

## Two-domain rule

URLs are built ONLY from `app_base_url` / `pages_base_url` in config — never
from `r.Host`. The session cookie is host-only and exists only on the pages
domain; the app-domain API is bearer-auth only. Do not "simplify" this into a
single domain: it isolates attacker-controlled report HTML from the API.

## Auth architecture (do not merge these)

- Human web login (`/auth/*`, pages domain): **goth** authorization-code flow.
- CLI login: RFC 8628 **device flow** in `internal/client/device.go` (x/oauth2).
- Server-side bearer validation in `internal/auth/validator_*.go`: OIDC =
  JWT via JWKS (go-oidc); GitHub = `GET /user` API call with 60s cache.
- Allowlist (`Match`: email then login) is applied after both flows.
- OIDC clients send the **id_token** as bearer; GitHub clients the access token.

## Commits

Use the [Conventional Commits](https://www.conventionalcommits.org/) format
(`feat:`, `fix:`, `docs:`, `chore:`, …). Write commit messages with the
`/caveman:caveman-commit` skill.

## Commands

```sh
go build ./...            # build everything
go test ./...             # run tests
buf lint && buf generate  # regenerate API code
pre-commit run --all-files
docker compose up --build # full stack behind Caddy
```

Local dev: set `dev: true` in config, use `app.localhost` / `pages.localhost`
(Caddy issues self-signed certs; see the commented block in `Caddyfile`).
